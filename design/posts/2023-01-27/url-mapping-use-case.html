GCWeb URL Mapping Use Case: MVA landing page

GCWeb use cases in the My VAC Account sign-in and register redesign

The problem

Users who click a 'sign-in' or 'register' button arrive on the same web page. Once there, they choose one of two third-party authentication services. The content of that page needs to address both returning and new users. It's a challenge to deliver straightforward content.

How it affects users

The application isn't responding to the choices that you've made. Users lose trust and confidence.

The solution

It is clearer to arrive on a page and read either 'sign-in' or 'register' when you press either button. Rather than create many pages with different content, we can use GCWeb's URL Mapping. We tweak the content on the page to fit people's intent

URL Mapping

Query strings are a way to add information to a URL. You may have noticed links that end in something like `?value=true`. Query strings can help us change the page's appearance, for example.

The benefit of this tool being a part of the GCWeb theme is that we don't create and maintain custom javascript. This tool makes reacting to query strings part of the HTML markup.

We will show and hide words on the landing page in the heading. The page will read either "Sign in to My VAC Account" or "Register for My VAC Account." We will also need the default content "Sign in to or Register for My VAC Account."

Default content

The default content is essential for two reasons. First, the content will not change if a user arrives at the URL from another source. Also, if the user disabled javascript, the content does not change. Second, this is the content that displays before the Web Experience Toolkit is ready. For people with slower connections, this could take a few seconds. This means users might read part way down the page before the content changes. The content should only change in a minor way.

This means the default content is visible unless either 'sign-in' or 'register' is in the query string. The register and sign-in content are hidden unless their respective query strings are present.

Sign-in and register states

When a user clicks a sign-in or register call to action, we tweak the content in several places:

  • Page heading: specifies either Sign-in or Register
  • Sign-in services descriptions: specifies either Sign-in or Register
  • Link to register or sign-in: Links to the opposite state, in case a user arrives here by mistake
HTML for h1 example 
                
    <!-- Landing page heading-->
    <div class="container">
        <!-- h1 with URLMapping on spans-->
        <h1 id="wb-cont" property="name">
        <!-- Show Sign in content when url?sign-in -->
        <span id="sign-in-h1-span" class="hidden"
                    data-wb-urlmapping={'
                    "sign-in": { 
                    "action": "removeClass", 
                    "source": "#sign-in-h1-span", 
                    "class": "hidden" }
                    }'>
                    Sign in to </span>
        <!-- Show Register content when url?register -->
        <span id="register-h1-span" class="hidden"
                    data-wb-urlmapping='{
                    "register": { 
                    "action": "removeClass", 
                    "source": "#register-h1-span", 
                    "class": "hidden" }
                    }'>
                    Register for </span>
    
        <!-- Hide default when url?register or url?sign-in -->
        <span id="default-h1-span" data-wb-urlmapping='{
                    "sign-in": { 
                    "action": "addClass", 
                    "source": "#default-h1-span", 
                    "class": "hidden" },
                    "register": { 
                        "action": "addClass", 
                        "source": "#default-h1-span", 
                        "class": "hidden" }
                    }'>
                    <!--Default empty ?-->
                    Sign in to or register for
                </span>
        My&nbsp;<abbr title="Veterans Affairs Canada">VAC</abbr>&nbsp;Account
        </h1>
